Serial Peripheral Interface (SPI)

SPI is an interface used to connect external chips to a microcontroller. A huge variety of chips use SPI including sensors, displays, external memory, networking devices, control devices, etc. It’s a four-wire synchronous protocol with one master and any number of slaves. SPI is capable of full-duplex mode (receiving and transmitting data at the same time) and can have higher throughputs than other related protocols like I2C. However, SPI will also often occupy more pins on the master/slave IC’s than related protocols.

A SPI bus consists of the following four data lines:
MOSI (Master Out Slave In): data line produced by master and received by slaves
MISO (Master In Slave Out): data line produced by slave but its generation is initiated by the master
SCK (Serial Clock): synchronizing clock generated by master
SS (Slave Select): select slave chip to communicate with (active LOW). The SPI master must have a separate slave select line for each slave.

The PIC32 can be in master or slave mode. This means that in master mode, the SDOx (serial data out) pin is MOSI and the SDIx (serial data in) pin is MISO. In slave mode, the reverse is true. SDOx pin is MISO and SDIx is MOSI.

When configuring the SPI pins, peripheral pin select handles the SDOx and SDIx pins. The slave select (SS) wire is dealt with manually as a general purpose I/O line. The Serial Clock (SCK) line is the only pin hardwired on PIC32MX220 microcontrollers. PIC32MX220’s have two SPI modules and the SCKx line for both of these is boxed in the pinout below.

All SPI devices, masters and slaves, contain a single serial shift register. A SPI data transfer works as follow:

  1. The master selects a slave by setting that slave’s SS pin LOW
  2. The master writes data to be transferred to its SPI shift register
  3. The serial clock becomes active
  4. In synchronicity with the clock, data will be pushed out of the master’s shift register onto the MOSI line and into the slave’s shift register at the same time that data is pushed out of the slaves shift register onto the MISO line and into the master’s shift register. That is, as a bit is pushed out of either the master or slave’s shift register, a new bit is pushed in. The shift registers never really empty, they just exchange their contents.
  5. The master either releases the chip by bring the SS pin HIGH or begins another data transfer

Since the slave chip will generally need time to perform some action based on the received data, SPI writes and reads are often split up. Generally, during a SPI write the master sends out a command and ignores the incoming message. During a SPI read, the master sends out a dummy byte and then reads the contents of its SPI shift register. When programming, the same function can be used to transmit and receive SPI data!

If we were doing 8-bit SPI transfers using the PIC32’s SPI1 module, the following function is able to transmit and receive data.

char SPI1_transfer( char b)
{
    SPI1BUF = b;                     // write to shift register to begin transmission
    while( !SPI1STATbits.SPIRBF);    // wait for transfer to complete
    return SPI1BUF;                  // read the shift register value
}

char SPI1_transfer( char b) { SPI1BUF = b; // write to shift register to begin transmission while( !SPI1STATbits.SPIRBF); // wait for transfer to complete return SPI1BUF; // read the shift register value }

Here, the SPI1BUF is the SFR that is directly associated with the SPI1 shift register. This means we would write to and read from the SPIxBUF register. The bit SPIRBF in the SFR SPIxSTAT is the “SPI Receive Buffer Full Status Bit”. This bit is cleared if the receive buffer is not full and set if the receive buffer is full. Since there is only one SPI shift register, data is being shifted out while data is coming in and the receive buffer is full when a transfer/receive operation is complete.

The following demonstrates how you could use this function, SPI1_transfer(), to send a data request to a slave chip and then read it’s response. Since the PIC32 doesn’t automatically deal with the slave select line, we’ll use pin RB4 and write to it manually. We’re assuming the slave select line can remain LOW during the entire TX/RX operation which is generally the case.

LATBbits.LATB4 = 0;                  // select slave device
SPI1_transfer( tx_data);             // transmit byte tx_data
char response = SPI1_transfer( 0);   // send dummy byte and read response
LATBbits.LATB4 = 1;                  // release slave device

LATBbits.LATB4 = 0; // select slave device SPI1_transfer( tx_data); // transmit byte tx_data char response = SPI1_transfer( 0); // send dummy byte and read response LATBbits.LATB4 = 1; // release slave device

SPI Configuration

The SFR SPIxCON does most of the standard SPI configuration while the register SPIxCON2 deals with special SPI interrupts and SPI audio capabilities (can implement I2S, left/right justified audio, or PCM).

Important SPIxCON bits for basic functionality include:
bit 15, ON: SPI on bit
bit 8, CKE: Clock Edge Select bit
bit 6, CKP: Clock Polarity Select bit
bit 5, MSTEN: Master Mode Enable bit

The ON and MSTEN bits are self explanatory. The CKP bit selects whether the clock (SCK) is LOW or HIGH when no SPI data is being transmitted (when idle). The CKE bit configures whether the SPI data will be read on a HIGH to LOW clock transition or a LOW to HIGH transition.

The SPIxCON register also contains bits to configure something called frame mode. I won’t go into details here but frame mode allows the slave select pin to be controlled and synchronized by the SPI module for precision timing events.

The SPIxBRG SFR configures the baud rate of the serial clock. Assuming the clock used is the peripheral bus clock (Pbclk), the baud rate is calculated by:

Baud = Frequency_Pbclk / ( 2*( SPIxBRG+1))

The following example configures SPI to be in master mode, have the serial clock idle LOW, receive data when the clock transitions from LOW to HIGH, and turn on. Assuming Pbclk is set to be the same frequency as SYSCLK running at 40MHz (true for most of the M5 tutorials), the baud rate will be configured to 4MHz.

// CKP (clock polarity control) = 0
// CKE (clock edge control) = 1
// 8-bit, Master Mode
// Baud = 4MHz = Fpb/( 2*( SPI1BRG+1)) = 40MHz/( 2*( 4+1))
SPI1CON = 0x8120;                   
SPI1BRG = 0x0004;

// CKP (clock polarity control) = 0 // CKE (clock edge control) = 1 // 8-bit, Master Mode // Baud = 4MHz = Fpb/( 2*( SPI1BRG+1)) = 40MHz/( 2*( 4+1)) SPI1CON = 0x8120; SPI1BRG = 0x0004;

The logic snapshot below uses these configurations. The master sends out a value of ‘5’ and gets a response of ‘3’ before the SS line is brought HIGH. The arrows on the serial clock line (SCK), indicate when a bit is being sampled. In this case from the LOW to HIGH transition.
